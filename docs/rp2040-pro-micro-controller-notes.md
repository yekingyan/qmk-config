# RP2040 Pro Micro 兼容主控记录

> 用于 `Ferris Sweep` / `QMK` 排障的硬件备忘。当前对象为用户实拍的 `RP2040` + `Pro Micro` 兼容主控。

## 结论先看

这块主控虽然采用了 `Pro Micro` 兼容排针命名，但底层 MCU 是 `RP2040`，因此部分引脚的上电行为不能按 AVR 时代的经验直接套用。

本次排障最关键的事实是：

- `F7 -> GP26`
- `F6 -> GP27`
- `F5 -> GP28`
- `F4 -> GP29`

也就是说，`Pro Micro` 别名里的 `F4~F7`，在这块板上实际落到了 `RP2040` 的 `GP26~GP29`，而这 4 个脚正好是 **ADC 复用脚**。

对 `Ferris Sweep` 这种 **direct pin** 键盘来说，如果矩阵/直连扫描使用了这组脚，必须额外关注：

- `FUNCSEL` 是否切到 `SIO`
- 输入使能 `IE` 是否打开
- 上拉 `PUE` 是否生效

否则就可能出现浮空误判、自动触发、ghost keys 等问题。

## 与当前 QMK 问题的关系

当前 `qmk-config` 项目中，`Ferris Sweep` 刷机后出现自动输出 `werttttt` 一类异常，排障主线已经集中到这组引脚上。

问题链条如下：

1. `Ferris Sweep` 使用 `direct pin`
2. 顶排部分按键映射到 `F7/F6/F5/F4`
3. 在该 RP2040 兼容主控上，这四脚实际为 `GP26~GP29`
4. `GP26~GP29` 是 ADC 复用脚
5. 若启动阶段未正确切回 `SIO` / 未开启输入使能 / 上拉未生效，就可能发生浮空误读
6. 最终表现为 ghost keys / 自动按键

## 当前判断

当前更像是 **RP2040 ADC/SIO 初始化问题**，而不是单纯的 `converter` 名字选错。

这意味着：

- `helios`
- `rp2040_ce`
- `sparkfun_pm2040`

这些 `converter` 差异可能影响部分外围定义，但对当前异常来说，**主线仍是 `GP26~GP29` 的初始化行为**。

## 写固件时的注意点

如果后续继续在 QMK 中使用这块主控：

- 不要把 `F4~F7` 当作“普通 AVR 风格 IO”看待
- 需要明确检查 `GP26~GP29` 在启动后是否处于键盘扫描所需状态
- 必要时在更早的初始化阶段处理，而不是仅靠用户层初始化钩子
- 若软件修复仍不稳定，再评估是否需要外部上拉电阻
